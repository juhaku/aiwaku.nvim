## 🔴 Critical (new)

### `terminal.lua` — `return 0` inside `nvim_buf_call` callback doesn't exit the outer function
`open_in_new_terminal_buf` wraps `jobstart` inside `vim.api.nvim_buf_call(new_buf, function() ... return 0 end)`. That `return 0` only exits the anonymous callback — `new_buf` is still returned even when the job failed, with no `terminal_job_id` set. The broken buffer gets cached in `state.session_bufnrs` and every subsequent operation fails with cryptic errors. Fix: after the `nvim_buf_call` block, check `vim.b[new_buf].terminal_job_id`; if nil, delete the buffer and return 0.

---

## 🟠 High (new)

### `session.lua` — async.void functions mutate shared state with no mutual exclusion
`toggle`, `select_session`, `clear_context`, `rename_session` all yield to the event loop mid-execution (during tmux jobs and `vim.ui.*` prompts). Rapid repeated keymap triggers can land two coroutines past the same guard check simultaneously, creating orphaned windows, double-opens, or corrupted `session_bufnrs`. Add a `state.busy` flag checked and set at the entry of each `async.void` function.

---

## 🟡 Medium (still present / new)

### `selection.lua:55-56` — `vim.b[bufnr]` accessed without nil-guarding `bufnr`
`state.session_bufnrs[session_name]` can return nil if `open_session` failed or a race occurred. Indexing `vim.b[nil]` crashes. Add a `bufnr` validity check before accessing `terminal_job_id`.

---

## 🟢 Low (still present)

### `terminal.lua:17` — `vim.tbl_deep_extend` for flat scalar tables
`bopts` and `{ desc = keymap.description }` are both flat (no nested tables). `vim.tbl_deep_extend` recursively traverses for no benefit. Use `vim.tbl_extend("force", ...)` instead. *(This same pattern was flagged in `init.lua` in review-1 and fixed there, but the same issue remains in `terminal.lua`.)*
