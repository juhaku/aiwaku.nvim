# Code Review — aiwaku.nvim

## 🔴 Critical

### `config.lua` — hardcoded user path
The default `cmd` is almost certainly set to `/home/juha/.local/bin/copilot`. That will break for every other user. Should be `{ "copilot" }` and resolved from `PATH`.

---

## 🟠 High

### `terminal.lua:40` — `jobstart` return value not validated
`vim.fn.jobstart()` returns `0` (cmd not executable) or `-1` (invalid args) on failure. Neither is checked. The bad job ID gets cached and `nvim_chan_send` with it will error. Add: `if job <= 0 then vim.notify(...) return 0 end`.

### `terminal.lua:40` + `session.lua:168` — orphaned processes on buffer delete
`clear_context` deletes the terminal buffer with `force = true` without calling `vim.fn.jobstop(job_id)` first. The tmux-attached process keeps running in the background. Stop the job before deleting the buffer.

### `tmux.lua:34-35` — `tonumber()` result not nil-checked
`get_session_created` does `os.date(config.date_format, tonumber(vim.trim(raw)))` without checking if `tonumber` returned nil. Same pattern on line 22 in `list_sessions`. If tmux returns empty/unexpected output, this crashes with a bad argument to `os.date`.

---

## 🟡 Medium

### `tmux.lua` — all `vim.fn.system` / `vim.fn.systemlist` calls block the event loop
Neovim 0.10+ provides `vim.system()` which is non-blocking. Every tmux shell-out (`session_exists`, `list_sessions`, `get_session_created`, `kill_session`, `rename_session`) stalls the UI. These should use `vim.system({ "tmux", ... }, { text = true })` with a synchronous wait (`:wait()`) or be restructured as async.

### `terminal.lua:9` — redundant/racy double API call in `buf_alive`
`nvim_buf_is_valid` + `nvim_buf_is_loaded` are two sequential API calls. `nvim_buf_is_valid` is sufficient to know a buffer is usable for most purposes; `is_loaded` only matters if you're about to read lines from it. The loaded check could be removed to save an API round-trip, or the whole function wrapped in `pcall` to guard against the rare TOCTOU window.

### `tmux.lua:61` — `cmd` not shell-escaped in `new_session_cmd`
`name` is `shellescape`d but `cmd` is concatenated raw. If `cmd` ever contains spaces or special characters (e.g. a path with spaces), the shell invocation breaks. Either `shellescape` each token or build the arg list as an array for `vim.system`.

---

## 🟢 Low / Suggestions

### `init.lua:27` — `vim.tbl_deep_extend` inside keymap loop
Every iteration allocates a new table for `{ desc = k.description }` and deep-merges it. Since all merged keys are top-level scalars, `vim.tbl_extend("force", kopts, { desc = k.description })` (shallow) is sufficient and cheaper. No need for deep here.

### `session.lua:67` — type annotation lying about return type
`@return Aiwaku.Session session` but the early-return path returns `nil`. The `---@diagnostic disable-next-line` is a workaround for a wrong annotation. Change to `@return Aiwaku.Session|nil`.

### `selection.lua:55` — redundant `buf_alive` check
`open_session` (called on line 51) already validates and/or creates the buffer. Checking `buf_alive` again immediately after is redundant in the happy path. Not harmful, but unnecessary work.
